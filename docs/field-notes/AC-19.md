# AC-19 — user-visible payload before send · field notes

*Surfaced by the **Thunderbird** evaluation (2026-07-09; comm-central trunk 153.0a1 @
`72b8ba076`) — a Mode-2 goal-impact read of a mail client that passes AC-19's send-time
review cleanly while the composition has already egressed. The AC:
[`spec/PNA_Spec.md` § Universal architectural commitments](../../spec/PNA_Spec.md#ac-19);
its sub-contract forms: WS-10 / CO-5.*

## Lesson

**Send-time review does not cover pre-send persistence.** AC-19 gates the moment a
communication *launches* — the user sees recipients, body, and merged data, then
confirms. But a workspace can persist the in-progress composition somewhere remote
*before* that gate is ever reached, and the payload has then left the device without any
review at all. Thunderbird is the worked example: the compose window is a clean AC-19
pass (nothing sends without the user pressing Send on the full composition), yet drafts
autosave every 5 minutes (`mailnews.js:260-262` in comm-central) to the account's Drafts
folder — which for an IMAP account is **the server**. The send-time seam held; the
private composition egressed anyway, through a durability feature.

The general shape: AC-19 is a gate on the *communication path*; pre-send copies travel
the *persistence path*, and only [AC-1's sealing facet](../../spec/PNA_Spec.md#ac-1)
(private data MUST NOT egress except through an explicit user action) governs that. The
two ACs compose — a candidate needs both read together. An evaluator who checks only the
compose/confirm flow will pass an app that leaks every keystroke of a half-written
message to a server.

## Negative invariants

- **A staged/in-progress composition must NOT be persisted off-device before the AC-19
  confirmation** — autosave targets, "cloud drafts," collaborative-editing buffers, and
  crash-recovery snapshots are all the same vector. Pin with a negative test: compose
  against a mock remote store, never press Send, assert zero remote writes.
- **Cancel must mean cancel:** after the user cancels at the AC-19 gate, no partial
  payload may remain in any off-device location it was staged to. (A server-side draft
  that outlives a cancelled send is a silent disclosure.)

## Pitfalls / look hardest at

- **Autosave and crash-recovery paths** — grep for timer-driven persistence in the
  composer, then trace where the store actually lives (local file vs. account server).
  The feature reads as Goal-4 durability; on a remote account it is Goal-3 egress.
- **The deterministic tier does not see this**: an egress lint keyed on network-call
  sites will attribute the write to a generic storage/sync layer, not to the composer —
  judge it architecturally.
- **In Mode-2 (adjacent-app) reads**, credit AC-19 honestly when the send gate is real,
  and land the pre-send egress on AC-1's sealing facet — the statuses stay precise and
  the Goal-3 note carries the composition of the two.
