## Branch / PR strategy

Each normative addition is **demonstrator-gated** (CONTRIBUTING: new AC/Exception/sub-contract needs
a green demonstrator — all three B riders and the D promotions have one). Recommended shape:

- **PR 1 — B (the #64 riders)**, normative + CI-gated; its own careful PR (may split B1/B2/B3 if review prefers).
- **PR 2 — D (exceptions-hardening promotion)**, normative + CI-gated.
- **PR 3 — the cut** (mechanical: VERSION, stamps, draft removal, CHANGELOG, tag) — *last*, after B + D land.

B and D could ride one "v0.2 normative batch" branch if preferred; keep the **cut** separate so the
version bump is a clean, reviewable, mostly-mechanical diff. Every PR: `just ci` green;
`docs/users-guide.md` updated in the same PR for any developer-visible change (the docs-current rule);
a fault-injection self-test for every new lint check (CLAUDE.md); definition-before-first-use honored
for any new term/AC.

## Phase 1 — B: formalize the three demonstrated #64 riders

### B1 — Distribution-axis verifiability split *(demo: PRM, build-from-source)*
- **What:** `never-distributed-single-user` today collapses *build-from-verifiable-source* with
  *opaque-binary* delivery; they differ in **independent verifiability**. Split them (rename/add a
  pick, or add a verifiability sub-dimension) in [`spec/axes.md` § Distribution](../spec/axes.md#distribution).
- **Encode:** the spec diff in `axes.md`; if the pick name changes, update PRM's `[flavor].distribution`
  in `reference_designs/prm/design.toml` + its `Architecture.md` flavor row, then re-sync the bundled
  copy (`just rearchive prm <ref> ~/src/prm`) and regenerate the index (`just realization-index`).
- **CI/scope:** new axis-pick vocabulary → a `lint_selftest` case (a `[flavor]` pick that resolves vs
  one that doesn't, already a pattern). **Decision to confirm:** rename the pick vs add a dimension —
  the rename ripples into PRM's flavor declaration.

### B2 — Safe-AI-write tier `AC-PRM-E/F` (review-required only) *(demo: PRM propose→review→apply)*
- **What:** add the **review-required** AI-write tier as normative flavor-derived ACs (`AC-PRM-E`,
  `AC-PRM-F`): no MCP/AI path commits a review-required write directly; the proposer stages, the
  workspace disposes. **Append-only / free-write tiers stay deferred** to PRM v0.2 — record that as an
  honest deferral (a § Scope note or RFC tag), not silently.
- **Encode:** new flavor-derived AC rows in [`spec/axes.md`](../spec/axes.md) under the triggering pick
  (MCP-exposure / AI-write surface — **decision:** which axis triggers them); contract `Realizes:`
  headers if a contract realizes them; PRM `Architecture.md` gains `AC-PRM-E/F` attestation rows
  (PRM already implements the loop — `mcp_servers/dedup_ops.py` propose-only + `core/apply.py`).
- **CI/scope:** the AC-ID traceability lint (`lint-spec-ids.py`) picks up new IDs automatically; add a
  `lint_selftest` case if a new check is introduced. Re-sync PRM bundled copy + index.

### B3 — User-mediation mechanism `UM-1/2/3` *(demo: fellows egress + PRM mutation)*
- **What:** promote user-mediation from "(proposed; UM-1/2/3)" in `exceptions.md` to a **normative
  mechanism**: the *actuation* boundary — proposer stages, principal disposes; **UM-1 no-bypass**
  (enforced at the data layer, not UI-only), **UM-2 separation**, **UM-3 legibility**. The bounded
  claim (separation/legibility/attribution, **not** comprehension; *actuator*, not *present human*) is
  load-bearing and must be stated.
- **Encode:** define the mechanism once — **decision:** a new `### User-mediation` section in
  `spec/PNA_Spec.md` (alongside the Goal/AC machinery), or its own `spec/user_mediation.md` paralleling
  `exceptions.md` / `constraints.md`. Both reference designs **already attest** UM-1/2/3 in their
  `Architecture.md` (fellows: the mediated-boundary registry; PRM: propose→review→apply), so the
  demonstrators are in place — wire the spec to them.
- **CI/scope:** if `UM-*` gets its own ID family + lint (like `EX-*`/`CST-*`), add the check **and** its
  fault-injection self-test in the same change. Update `users-guide.md` (a new mechanism a builder sees).

## Phase 2 — D: exceptions-hardening (RFC → normative, where demonstrated)

- **What:** the three `(Proposed, RFC)` amendments in [`spec/exceptions.md`](../spec/exceptions.md):
  (1) the **`pna-active`** predicate split (the mode bit a relying party keys on, vs. exception-handling
  conformance); (2) **EX-H7 fail-closed** (consent-to-human propagation); (3) the **un-relaxable floor**.
  Promote each to normative **only where a demonstrator supports it** (fellows ships the `EX-CLOUD-LLM`
  handler + the `<body data-pna-mode>` machine-readable signal — strong support for `pna-active` + the
  handler discipline); keep any unsupported part RFC-tagged rather than over-promoting.
- **Encode:** drop the `(Proposed, RFC)` tags on the promoted parts in `exceptions.md`; update the
  cross-references in `spec/PNA_Spec.md` that currently say "proposed RFC" / "proposed predicate split"
  (§ Vocabulary `pna-active`, § Composition, § Vision) to normative; reflect in `SKILL.md`'s evaluate
  flow if the predicate becomes a checked output.
- **CI/scope:** **delicate** — normative wording over a machine-parsed file. Any new/locked check (e.g.
  a `pna-active` predicate lint) needs a fault-injection self-test. Confirm the fellows `Architecture.md`
  attestation already covers the promoted handler clauses (it attests EX-H1–H6/H8 conformant, H7
  best-effort) → no design goes non-conformant on promotion. **Decision:** promote all three, or only
  `pna-active` + EX-H7 and hold the un-relaxable floor as RFC?
